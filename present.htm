Problem Statement

Challenge

Migrating Tableau dashboards to Looker is manual, time-consuming, and error-prone. Large datasources with 350+ fields exceed GenAI context windows.

Requirements

  • Automated conversion of Tableau XML to Looker LookML
  • Preserve semantic meaning and business logic
  • Handle complex datasources with hundreds of fields
  • Maintain field relationships and formulas
  • Scalable architecture for production use

The Approach

Send entire Tableau datasource XML directly to GenAI for conversion to DSL format, then convert DSL to LookML.

✓ Advantages

  • Simple architecture
  • Single-pass processing
  • Works for small datasources

✗ Critical Issues

  • Context window overflow - 350+ fields exceed token limits
  • No scalability - Can't handle real-world datasources
  • All-or-nothing - Entire process fails if too large
  • Incomplete parsing - Large XMLs get truncated

The Solution

Hybrid Architecture: Minimal deterministic code for XML parsing + GenAI for semantic understanding + Modular chunking for scalability

Key Innovation

Separation of Concerns: Code handles mechanical extraction, GenAI handles intelligent interpretation. Process 100 fields at a time, not 350 all at once.

Architecture Overview

Layer 1: XML Parsing (Deterministic Code)

• Extract XML structure to clean JSON
• No interpretation, just transformation
• Pair metadata records with column definitions
• Preserve all field relationships

Layer 2: Chunking (Deterministic Code)

• Split fields into 100-field chunks
• Connection chunk (1x) - tables, joins, database info
• Field chunks (Nx) - dimensions, measures, calculations
• Each chunk is self-contained

Layer 3: GenAI Processing (Semantic Layer)

• Process each chunk with specialized prompts
• Extract semantic meaning and business logic
• Normalize formulas and field references
• Generate DSL with complete field mapping

Layer 4: DSL to LookML (GenAI + MCP)

• Convert DSL to LookML syntax
• Deploy to Looker via MCP tools
• Validate and test generated views/explores

DSL (Domain Specific Language)

Purpose

Compact intermediate format that preserves all semantic meaning while being platform-agnostic. Acts as a bridge between Tableau and Looker.

Format Example

DS:fct_metrichomepage | CONN:bigquery.analytics.tableau_to_bigquery TABLES: fct_metrichomepageall_fac_day|type:table dim_location_hbuop|type:table JOINS: fct_metrichomepageall_fac_day.facility_code = dim_location_hbuop.facility_code|type:left DIMS: facility_code|table:"fct_metrichomepageall_fac_day"|db:"FacilityCode"|label:"Facility Code"|string order_date|table:"fact_orders"|db:"Order_Date"|label:"Date Ordered"|date MEASURES: sales|table:"fact_orders"|db:"Sales"|number profit|table:"fact_orders"|db:"Profit"|number CALCS: total_revenue|label:"Total Revenue"|measure|SUM(sales) profit_margin|label:"Profit %"|measure|(profit / sales) * 100

Code Organization

Key Design Principles

  • Separation of Concerns: Code does extraction, GenAI does interpretation
  • Modularity: Each component has single responsibility
  • No Processing in Parsing: Pure XML → JSON transformation
  • Context Preservation: All field relationships maintained
  • Self-Contained Chunks: Each chunk processable independently

Technical Advantages

Scalability

Linear Scaling: Processing time grows linearly with datasource size. 1000 fields = 10 chunks @ 100 fields each. Predictable and manageable.

Reliability

Error Isolation

  • Chunk-level failures don't break entire process
  • Can retry individual chunks
  • Partial success possible

Validation Points

  • Validate after each phase
  • Catch errors early
  • Clear failure points

Maintainability

  • Modular Design: Each component independently testable
  • Clear Boundaries: Code vs GenAI responsibilities well-defined
  • Extensible: Easy to add new Tableau features or Looker capabilities
  • Debuggable: Can inspect intermediate outputs at each stage
